# Modernization Handoff

Living status doc for the node-wpapi modernization. See `modernization-plan.md`
(vision) and `modernization-execution-plan.md` (phased plan). Trunk is `main`;
each phase lands via a `--no-ff` merge.

## Current state: Phase 1 complete (modern toolchain, still JS)

Full suite green against the local wp-env WordPress: `805 passed, 16 skipped,
0 failed` (39 suites) — identical to the Phase 0 baseline. No library
behavior changed; this phase swapped infrastructure only.

### What Phase 1 did

- **Build: webpack + Grunt + Babel → tsdown.** `tsdown.config.mts` builds
  three Node entries (`index` from `wpapi.js`, `fetch`, `superagent`) as dual
  ESM+CJS with generated `.d.ts`, plus two browser UMD bundles (`dist/browser/`)
  exposing the `WPAPI` global. `package.json`'s `main`/`module`/`types`/`exports`
  are now auto-generated by tsdown's `exports: true`.
  - Each Node entry is its OWN independent tsdown config (not one multi-entry
    config), because `fetch`/`superagent` both `require('../wpapi')`: sharing
    a chunk across entries breaks rolldown's CJS `module.exports` codegen
    (verified — it silently emits four sequential `module.exports = ...`
    assignments into one file, and the last one wins). Bundling each entry's
    full dependency closure independently avoids that, at the cost of some
    duplicated code across `dist/*.cjs`.
  - The browser UMD builds alias `node-fetch`/`form-data`/`fs` to thin stubs
    in `build/browser-shims/` (mirrors the old webpack `externals`/`node.fs`
    config). tsdown auto-externalizes anything in `package.json` deps —
    `deps.alwaysBundle` forces `node-fetch`/`form-data`/`qs`/`li`/`superagent`
    back into the browser bundle so the aliases and real bundling apply.
  - `.d.ts` generation needs `tsconfig.json`'s `module: "commonjs"` (source is
    plain CJS) and TypeScript's classic `tsc` generator — installing
    TypeScript unpinned resolves to the experimental `7.x` native-preview
    (`tsgo`), which tsdown auto-selects and which happens to tolerate a
    pattern the stable `tsc` generator rejects: a JSDoc `@returns {WPRequest}`
    forced `wpapi.d.ts` to cross-reference `lib/constructors/wp-request.js`'s
    type, and `rolldown-plugin-dts` failed the build trying to bundle that
    CJS-shaped declaration ("MISSING_EXPORT: default is not exported by
    lib/constructors/wp-request.js"). Fixed by widening those two JSDoc
    return types to `{Object}`, which avoids the crash — the richer,
    route-specific typing is Phase 3's job anyway. The build still logs
    "rolldown-plugin-dts ... does not support bundling CommonJS dts input"
    as a non-fatal warning for all three entries; that's expected given the
    source is CJS, and doesn't block `.d.ts` output. Keep TypeScript
    pinned to the stable 5.x line; don't let it drift back to a `7.x` preview.
- **Tests: Jest → Vitest.** `vitest.config.js` sets `globals: true` (Jest-
  compatible globals) and a setup file (`tests/helpers/vitest-jest-compat.js`)
  aliasing `jest` → `vi`, so none of the ~30 `jest.fn()`/`jest.spyOn()` call
  sites needed touching. One real fix: `tests/unit/wpapi.js` required the
  package root (`require('../../')`), which resolved through `package.json`
  `main` — now `dist/index.cjs` instead of source — causing an `instanceof`
  mismatch against the separately-required `WPRequest` class. Changed it to
  `require('../../wpapi')` directly; this also decouples that suite from
  whatever `main` happens to point to (relevant again once Phase 2 rebinds
  `main` to the fetch-bound export).
- **Lint/format: ESLint 4 + `.eslintrc.js` → ESLint 9 flat config + Prettier.**
  `eslint.config.js` carries forward the exact same custom rules (including
  the space-inside-parens/brackets/braces convention) — **by explicit
  decision, Prettier was added but NOT run across the existing codebase**,
  since Prettier can't express that spacing convention and would fight the
  ESLint rules on every file it touched. Prettier (`.prettierrc.json`, tabs +
  single quotes) is available via `npm run format`/`format:check` for new
  code or a future full-codebase decision; it isn't wired into `lint`/CI.
  `eslint --fix` also surfaced (and fixed) real `eslint:recommended` findings
  that predate this phase: two `hasOwnProperty` calls flagged by
  `no-prototype-builtins` (now `Object.prototype.hasOwnProperty.call(...)`)
  and one unused `catch (e)` binding (now an optional catch binding).
- **Integration test flakiness (Phase 0's predicted risk, now hit and fixed).**
  Vitest runs test files in parallel by default; the CRUD suites share one WP
  DB, and under load the media-upload test missed the 5s default timeout.
  `test:integration` now runs `vitest run tests/integration
  --no-file-parallelism` (the Vitest equivalent of the `--runInBand` Phase 0
  flagged), and `vitest.config.js` sets `testTimeout: 15000`.
- **`package.json`:** `engines.node: ">=18"`, `browserslist`/IE11 dropped,
  `typecheck` script added (`tsc --noEmit`; not yet meaningful — `checkJs` is
  off until Phase 3). Removed the `docs`/`zip` script's dependency on Grunt
  (kept `docs` running `jsdoc` directly); `predocs`/`postdocs`/`jekyll`/
  `release-docs` are untouched dead-ish code pending Phase 6's docs triage.
- **CI: `.travis.yml` → GitHub Actions.** `.github/workflows/ci.yml` has
  three jobs, not the plan's literal "18/20/latest matrix" — the toolchain
  itself doesn't support that range: `tsdown`/`rolldown` need Node >=22.18,
  `vitest` needs >=20. So: `test` (Node 22/24/latest — lint, typecheck,
  build, unit tests, uploads `dist/` as an artifact from the `latest` leg);
  `dist-smoke` (Node 18/20 — downloads that artifact and just requires the
  pre-built CJS/ESM entries, checking the *published* `engines.node: >=18`
  promise without needing the build tooling to run on old Node); `integration`
  (Node 22 — boots `@wordpress/env`, runs `test:integration`; note `wp-env
  logs` defaults to `--watch: true` and never exits, so the workflow calls it
  directly with `--no-watch` rather than via the `env:logs` dev script).
  `release.yml` is scaffolded per the plan (build/test/`npm publish
  --provenance` on a GitHub Release, Node 22) but inert until an `NPM_TOKEN`
  secret is added and publishing resumes.
  `package-lock.json` is now committed (`.gitignore`'s old "consumers
  maintain their own lockfile" rationale doesn't hold for CI reproducibility
  — `actions/setup-node`'s `cache: npm` and `npm ci` both need one checked in).
- Deleted: `webpack.config*.js`, `Gruntfile.js`, `build/grunt/`, all
  `.eslintrc.js`/`.eslintignore` files, `.travis.yml`, and the Babel/webpack/
  Grunt/Jest devDependencies.

### Known gap (deferred to Phase 5)

- **`discover` / live-route bootstrap fails on modern WP.** Unchanged from
  Phase 0 — see prior note below. Still `describe.skip`-ped.
  - The route-tree regex parser (`lib/route-tree.js:83`) throws "Unterminated
    group" on routes whose named groups contain nested patterns, e.g.
    `wp/v2/templates/(?P<id>...)`. Default-mode instances are unaffected.

## Dev workflow

```
npm install
npm run env:start     # boot wp-env (localhost:2747)
npm run env:seed      # once, against a fresh env
npm run lint          # eslint .
npm run typecheck     # tsc --noEmit
npm run build         # tsdown -> dist/ (ESM+CJS+UMD+.d.ts)
npm run test:unit     # vitest, no WP needed
npm run test:integration
```
